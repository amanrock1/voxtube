1. Understanding of the Product
================================================================================

What Problem the Product Solves:
YouTube creators with growing channels (10k to 10M+ subscribers) receive thousands of comments per video. Valuable feedback, video ideas, business opportunities, bug reports, and high-value fan interactions are drowned in a sea of spam, self-promotion, bot networks, hate speech, and low-effort comments. Creators lack the time to read everything, leading to missed audience engagement opportunities, lost sponsorship leads, and poor channel community management.

This platform automates comment processing to:
* Identify high-priority comments (business leads, product bugs, high-value fan feedback).
* Filter out noise (spam, bots, hate speech) with high accuracy.
* Generate actionable content ideas and audience demographic/psychographic insights from comment trends.

Target Users:
* Solo Creators (Mid-tier, 50k–500k subscribers): Need to identify top fans, filter spam, and quickly find video ideas.
* Creator Agencies & Managers (Managing 5M+ subscribers): Need multi-channel dashboards, toxicity reports, automated moderation queues, and brand safety compliance.
* Enterprise Brand Channels: Require deep sentiment analysis, PR risk alerts, and competitor comment comparisons.

Essential Features (V1 / MVP):
1. OAuth Integration & Safe Syncing: Securely fetch comment threads from YouTube API using background workers.
2. Basic Spam & Self-Promotion Filter: Rule-based and basic heuristic models to quarantine self-promotion links and repetitive spam.
3. Comment Categorization (Bug, Suggestion, Praise, Business Inquiry): Help creators find actionable comments instantly.
4. High-Level Sentiment Analytics: Aggregate channel and video-level sentiment over time.
5. Creator Dashboard: Simple, fast interface displaying key metrics and a searchable, categorized comment feed.

Unnecessary Features for V1:
* Real-time WebSocket Streaming: Standard polling or periodic webhook-triggered sync runs are sufficient and much cheaper.
* Deep Sarcasm & Cultural Context Analysis: Computationally expensive, low reliability, and not critical for initial launch.
* Fully Autonomous Auto-Replies: High risk of PR damage if the AI hallucinates or misinterprets a sensitive comment.
* Multi-Platform Integration (TikTok, Instagram, Twitch): Focus on winning YouTube first.

2. Feasibility Analysis
================================================================================

* Sentiment Analysis: [EASY]
  - Rationale: Pre-trained models (e.g., RoBERTa-sentiment) or managed APIs (Gemini, Claude) can reliably classify positive, negative, and neutral sentiment out-of-the-box.

* Comment Categorization: [EASY TO MEDIUM]
  - Rationale: Classifying comments into "Question", "Feature Request", "Business Inquiry", or "Praise" is highly viable using structured zero-shot prompting with lightweight LLMs (e.g., Gemini Flash, GPT-4o-mini).

* Spam Detection: [MEDIUM]
  - Rationale: Simple link/keyword filters are easy, but detecting stealthy self-promotion or copy-pasted comments requires heuristic scoring, text embeddings similarity checks, and rate-limiting rules.

* Audience Insight Generation: [MEDIUM]
  - Rationale: Synthesizing thousands of comments into a "Weekly Summary" or "Video Ideas list" is feasible via map-reduce aggregation chains feeding into an LLM context.

* Bot Detection: [HARD]
  - Rationale: Requires cross-referencing account creation dates, posting frequencies, username patterns, and network behavior. The YouTube API does not expose all required metadata, limiting access to user history.

* Sarcasm Detection: [HARD]
  - Rationale: Extremely challenging because sarcasm relies heavily on external context, video content, and internet subcultures. Zero-shot LLMs perform poorly here without video transcripts or context injection.

* Cultural Reference / Meme Understanding: [VERY HARD]
  - Rationale: Memes and slang evolve weekly. Keeping a static model updated with internet culture requires continuous fine-tuning or web-search-augmented retrieval (RAG), which is cost-prohibitive.

* Recommendation Engine (Smart Replies): [HARD]
  - Rationale: Draft generation is medium, but matching creator voice, maintaining brand safety, and handling complex queries requires fine-tuned models or complex vector-retrieval guardrails.

3. Architecture Proposal
================================================================================

Complete System Components:

[UI (React/Next.js)] -> [NGINX API Gateway] -> [API Server (NestJS)]
                                                |
                      +-------------------------+-------------------------+
                      |                                                   |
             [PostgreSQL (pgvector)]                                [Redis (Queue)]
                                                                          |
                                                                   [Celery Workers]
                                                                          |
                                                  +-----------------------+-----------------------+
                                                  |                       |                       |
                                           [YouTube API]             [Gemini/OpenAI]        [Local Models]

Architectural Decisions:

1. Backend Framework: Node.js (NestJS) vs. Python (FastAPI)
   - Decision: Node.js (NestJS) for the main API; Python for background workers handling AI tasks.
   - Why Chosen: NestJS provides structured, scalable, out-of-the-box support for TypeScript and microservices. Python is optimal for handling NLP, tokenization, and model integration.
   - Alternative: Single language (Pure Python FastAPI).
   - Pros: Clear separation of concerns, optimal language performance for tasks.
   - Cons: Multi-language setup increases tooling and repository management overhead.
   - Scalability Concerns: Node handles high concurrent API requests easily; Python handles compute-heavy processing asynchronously via workers.
   - Future Limitations: Context switching overhead for developers.

2. Message Queue & Task Scheduler: BullMQ vs. Celery
   - Decision: BullMQ (Redis-backed) for simplicity in V1.
   - Why Chosen: Minimizes infrastructure complexity by utilizing the Redis instance already needed for caching.
   - Alternative: RabbitMQ + Celery.
   - Pros: Very easy to set up and monitor; high throughput for simple tasks.
   - Cons: Storing massive message payloads can cause Redis RAM utilization spikes.
   - Scalability Concerns: Redis scales vertically; could become a cost bottleneck.
   - Future Limitations: High volume requires migrating to RabbitMQ/Kafka.

3. Primary Database: PostgreSQL (with pgvector)
   - Decision: PostgreSQL.
   - Why Chosen: Relational integrity, ACID compliance, and pgvector allows semantic searches without an external database.
   - Alternative: MongoDB or Cassandra.
   - Pros: Solid support, single database cluster to manage.
   - Cons: Scaling write-heavy spikes requires partitioning and replication strategies.
   - Scalability Concerns: Inserting millions of comments quickly can lock tables.
   - Future Limitations: Must implement horizontal partitioning once DB goes past ~100GB.

4. AI Architecture
================================================================================

* Spam Detection:
  - How it works: A local regex/blacklist check runs first to catch standard links/phrases. Then, comment text is transformed to vector embeddings and compared via cosine similarity against a database of known spam templates.

* Sentiment Analysis:
  - How it works: Runs locally/serverless using a pre-trained RoBERTa model fine-tuned for sentiment analysis. This avoids external API call charges.

* Comment Categorization:
  - How it works: Comments are batched (e.g., groups of 50-100) and sent to a cheap, fast model (like Gemini Flash) with structured system instructions to classify and tag categories.

* Bot Detection:
  - How it works: Compares comment posting patterns, repetition frequency, profile name randomness, and cross-channel duplication. Accounts with highly repetitive or speed-anomalous postings are flagged.

* Recommendation Engine:
  - How it works: RAG pipeline. Comments are embedded, compared against a vector library of past creator replies, and passed to a generative model with instructions to match the creator's writing style.

* Sarcasm Detection:
  - How it works: Only triggered when a comment has contrasting sentiment scores (e.g., highly positive words but negative feedback history) or when high accuracy is requested. References video transcript to evaluate context mismatch.

* Cultural Context Understanding:
  - How it works: Uses a modern LLM with access to search or a weekly updated database of internet slang and cultural memes to interpret slang-heavy comments.

5. Biggest Technical Risks
================================================================================

1. YouTube API Quota Exhaustion:
   - What fails: Syncing comments fails once the daily quota (10,000 units) is exhausted.
   - Realistic Expectations: A single viral video sync can consume the entire daily quota. Quota extensions take weeks to approve. Smart syncing and caching are mandatory.

2. Cost of LLM Processing:
   - What fails: Gross profit margins. Running LLM prompts on every incoming comment will make the product unprofitable.
   - Realistic Expectations: Must implement local filters (regex, semantic embeddings) to drop noise before invoking LLMs.

3. Sarcasm and Meme Detection Accuracy:
   - What fails: Sarcasm classification will have less than 60% accuracy out of context.
   - Realistic Expectations: Accept that accuracy will be imperfect. Provide UI tools for creators to quickly correct classification errors.

To build, deploy, and run this platform completely for free, we leverage the following developer-friendly free tiers:

   - Includes `pgvector` extension out of the box for semantic comment search.

3. AI Model Engine: Google AI Studio / Gemini API (Free Tier)
   - Gemini 1.5 Flash has a highly generous free rate limit (15 requests per minute / 1 million tokens per minute).
   - Ideal for categorization, spam detection, and summaries without paying a cent.

4. Backend Hosting: Render (Free Web Service Tier) or Supabase Edge Functions
   - Render provides free Node.js/Python hosting (note: server spins down after 15 minutes of inactivity).
   - Supabase Edge Functions (Deno/TypeScript) run serverless and don't suffer spin-down latency.

12. Form Factor Choice: SaaS Web App vs. Chrome Extension
================================================================================

Depending on how you want creators to interact with the tool, here are the two paths:

PATH A: Web Dashboard (SaaS Web App)
- How it works:
  1. Creator logs into your web app with Google (YouTube OAuth).
  2. Backend fetches comments via YouTube API and stores them in PostgreSQL.
  3. UI displays comments categorized with filters, graphs, and action items.
- Resume Impact: High. Demonstrates understanding of databases, OAuth, API integrations, and async background workers.
- Drawback: YouTube API quota limits (10k units/day) are strict, and backend background tasks are limited on free hosting.

PATH B: Chrome Extension
- How it works:
  1. Creator installs the extension and opens YouTube Studio (`studio.youtube.com/comments`).
  2. The extension reads the comments directly from the page DOM (meaning zero YouTube API quota cost!).
  3. The extension calls the Gemini API directly (via a secure serverless function) and overlays labels ("Bug", "Sponsor Lead", "Toxicity") directly on their YouTube screen.
- Resume Impact: High. Demonstrates DOM manipulation, chrome browser APIs, security, and smart UI injection.
- Drawback: Only works on desktop browsers (Chrome/Brave/Edge) and cannot run background scans when the tab is closed.

RECOMMENDED STEP 1 FOR V1:
We will start with PATH A (Web Dashboard) but design the APIs cleanly. This is the most structured way to learn full-stack web development.

14. Key Concepts You Need to Learn (For Resume & Interviews)
================================================================================

If an interviewer asks you about this project, you need to understand and explain these 4 key pillars:

1. OAuth 2.0 Flow:
   - Concept: How does Google securely log in your users and grant your app permission to read comments?
   - What to know: Understand Authorization Codes, Access Tokens (temporary), Refresh Tokens (stored securely in Supabase to fetch comments in the background), and Scopes (e.g., `https://www.googleapis.com/auth/youtube.readonly`).

2. Semantic Vector Search & pgvector:
   - Concept: Traditional search looks for exact keywords. Semantic search looks for meaning.
   - What to know: The comment text is converted into an array of numbers (embeddings) using a model. We calculate the mathematical distance (cosine similarity) between comment embeddings in PostgreSQL using `pgvector` to find similar comments.

3. AI Batching & Structured Output:
   - Concept: Sending 100 comments individually to Gemini is slow and expensive.
   - What to know: We bundle comments into a structured JSON array and instruct the LLM to output a matching JSON array with classifications. This reduces network roundtrips and optimizes API usage.

4. Asynchronous Task Queuing:
   - Concept: When a user requests to sync 5,000 comments, the HTTP request should not hang.
   - What to know: The API accepts the request and immediately replies "Sync started". In the background, a worker queue handles fetching pages of comments from Google and processing them with AI.

16. Detailed V1 Build Stages (Normal User Focus)
================================================================================

Here is the exact engineering breakdown of how we will build V1 step-by-step:

STAGE 1: Setup Workspace, DB, and API (Backend Foundation)
* Tasks:
  1. Initialize the monorepo using TypeScript for the backend (NestJS/Express) and frontend (React).
  2. Set up a Supabase project and define PostgreSQL tables:
     - `videos`: Store basic video metadata (ID, title, channel name, thumbnail).
     - `comments`: Store comment text, author, sentiment score, category tag, and foreign key to `videos`.
  3. Create DB indexes on comment sentiment, categories, and video IDs to make searching fast.

STAGE 2: YouTube Scraper & Sync Engine (Data Fetching)
* Tasks:
  1. Write a server-side service that takes a YouTube Video ID and requests comments using the Google API Client (`youtube.commentThreads.list`).
  2. Implement pagination to fetch up to 200–500 comments (an optimal limit for free tier storage and fast processing).
  3. Save the fetched comments and video metadata into PostgreSQL, ensuring we don't save duplicate comments if analyzed multiple times.

STAGE 3: Gemini AI Pipeline (Analysis Backend)
* Tasks:
  1. Setup Google AI Studio API credentials.
  2. Write batch processing functions:
     - Group comments in batches of 50.
     - Send comments to Gemini 1.5 Flash using structured prompting asking the model to return a JSON array containing the Sentiment (Positive, Neutral, Negative) and Category (Question, Feedback, Praise, Noise) for each comment.
  3. Write a video summarization service that sends the top comments to Gemini and requests a structured markdown summary.
  4. Write the results back to the database.

STAGE 4: Frontend Development (Dashboard & Smart Inbox UI)
* Tasks:
  1. Design a premium landing page with a central YouTube URL search bar and quick-test video cards.
  2. Build a Dashboard page using Recharts (or similar library) for visual charts (sentiment split, category ratios).
  3. Build the "Smart Comment Feed" containing:
     - Live keyword search.
     - Filter pill buttons (Questions, Bugs, Praise, Negative).
     - Highlighted AI Video Summary block.
  4. Add loading skeleton states (essential for premium UI while AI is processing the video).

STAGE 5: Seeding, Testing, and Free Deployment
* Tasks:
  1. Write a script to insert pre-coded mock videos and comments (with varied sentiments and categories) to test the UI instantly without API keys.
  2. Run system integration tests (pasting a real link -> scraping -> AI classifying -> rendering UI).
  3. Deploy the database on Supabase, backend on Render, and frontend on Vercel.
